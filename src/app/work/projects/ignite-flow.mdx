## Overview

[View on GitHub](https://github.com/JazzzFM/IgniteFlow)

Every data team eventually writes the same boilerplate: a Docker setup for Spark, some Terraform to provision the cluster, a CI/CD pipeline cobbled together under deadline pressure. Then they spend weeks debugging environment inconsistencies between development and production. IgniteFlow packages that hard-won infrastructure knowledge into an opinionated, production-ready template—so teams can skip the scaffolding and start on the actual problem.

## Design Principles

Three decisions shape the template's architecture:

1. **Container-first**: Spark jobs run inside Docker images from day one. Development, testing, and production share the same runtime, which eliminates the most common class of "works on my machine" failures before they happen.
2. **Infrastructure as Code**: Terraform/OpenTofu defines every cloud resource. The infrastructure is version-controlled, reviewable in pull requests, and reproducible with a single `terraform apply`. No manual console clicks, no configuration drift.
3. **MLOps from the start**: MLflow is wired into the template—experiment tracking, model registry, and deployment hooks are available without additional configuration. Teams don't have to retrofit observability later.

## What's Included

- Dockerized Spark environment with configurable worker replicas for local and cloud parity
- Terraform modules for AWS (EMR, S3, IAM) ready for production use
- GitHub Actions pipeline covering lint, test, and deploy stages with environment gates
- Example ETL job and ML training script structured to production conventions
- MLflow server configured with S3 artifact backend and PostgreSQL metadata store

## Outcome

IgniteFlow reduces the time from "new project" to "first Spark job running in the cloud" from days to hours. For data teams, that means more cycles on modeling and less on infrastructure archaeology—and a baseline that enforces reliability and observability by default rather than as an afterthought.
