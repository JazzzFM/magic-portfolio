## Overview

[View on GitHub](https://github.com/JazzzFM/SentinelMCP)

Most RAG systems answer questions from a fixed knowledge base. SentinelMCP executes tasks. It combines **AutoGen**'s multi-agent orchestration with the **Model Context Protocol (MCP)** to give agents real-world capabilities—live web browsing, dynamic data analysis, iterative peer review—coordinated through a structured conversation protocol rather than rigid directed graphs.

## The Problem with Static Agents

Retrieval-only pipelines hit a ceiling when tasks require multi-step reasoning, external data fetching, or self-validation. A single agent with tool access can fetch a web page; it struggles to plan a research sequence, critique its own findings, and produce a reliable synthesis. That is a coordination problem, not a retrieval problem. SentinelMCP addresses coordination directly.

## Architecture

The system is built on three layers:

**Agent layer (AutoGen Core)**

A `GroupChat` manager orchestrates a team of specialized agents, dynamically selecting which agent should act next based on the current task state:

- **Commander** — decomposes user intent into a structured task backlog.
- **Web Researcher** — executes live queries via the Playwright MCP server, handling JavaScript-heavy pages and extracting real-time data.
- **Critic** — validates the researcher's output for accuracy and completeness before the final response is delivered.

**Tool layer (MCP)**

All external capabilities are connected through MCP servers rather than custom tool integrations. This decouples tool implementation from agent logic entirely:

- **Playwright MCP** — autonomous web browsing with full JavaScript rendering, enabling agents to interact with live websites as a user would.
- **Tidewave MCP** — specialized data analysis and institutional tool connectivity for structured data domains.

Adding a new capability means plugging in a new MCP server—not modifying agent code.

**Observability layer**

Integrated tracing records each agent's deliberations and every tool call, making it possible to audit why a given decision was made. For production agentic systems, this is not optional—it is the difference between a system you can trust and one you can only hope works.

## Outcome

SentinelMCP demonstrates that the most consequential design decision in an agentic system isn't which model to use—it's how agents are coordinated and how tools are connected. By separating orchestration (AutoGen) from tool access (MCP), the architecture is modular, extensible, and auditable in ways that monolithic single-agent frameworks are not.
